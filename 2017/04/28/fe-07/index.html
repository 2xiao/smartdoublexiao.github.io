首先请弄清楚这些概念：

  • MVVM

    • 前端框架不止是js，更重要的还有设计思想和编程思维。

      而且最重要的是它不是一蹴而就的，是一个团队迭代无数次从最初的设想然后一步步踩坑写成的。

      从最开始的操作dom时代（以jq为主）到现在的数据驱动时代（三大框架），我相信保持这样的思维方式才会不断让自己得到提升：在用框架的时候多问问自己为什么要这样用？

      我目前最熟悉的是NG,我对NG这个框架的理解是这样的：

      为什么要组件化？为什么要MV*分离？为什么要数据驱动？为什么要使用服务？我在使用它的时候总会有一连串的问题冒出来。

      只知道用确不知道为什么用，用了有什么好处？有什么坏处？这大概就是我春招面试惨败的原因吧。

      就拿一个非常简单的NG指令*ngIf来说，不去深究它，只知道它可以根据表达式的真假决定是否显示一块HTML内容的话，就太悲哀了。

      去查阅相关资料后(包括但不限于书籍、官网文档、NG的github、stackoverflow),我发现可以自定义一个myIf指令来实现ngIf的功能：
      ​

      这时又涌现出一大串问题：

      1. 为什么要引入Directive？
        1. 为什么myIf既要和selector还要和Input关系？
        2. ngOnChanges方法又是干吗的？
        3. ViewContainerRef和TemplateRef又是什么鬼？为什么构造函数里只有它们？
        4. 这样做的好处和坏处是什么？

      这时继续查阅相关资料，知道了

      1. Directive是自定义指令的时候必须使用的,顺便还复习了一下装饰者模式,它的作用就是动态地扩展一个对象的功能。
        1. selector是用来确定装饰器目标的。Input可以让NG让数据从模板流向类，这样myIf绑定的表达式就进入到这个类了。
        2. ngOnChanges方法是NG自带的生命周期钩子之一，当检测到输入的数据变化时就会执行，涉及到模板方法,ngIf的核心逻辑就是这个方法里实现的，SimpleChange是其中一种变化类型。
        3. ViewContainerRef是管理DOM元素的，它内部的一些方法可以帮助动态维护DOM元素。TemplateRef就是NG中获取的DOM元素。在调用指令的时候和这个myIf绑定的HTML元素相关的ViewContainerRef和TemplateRef就被当成构造参数传入了这个类。
        4. 这样做就可以实现数据驱动了，前端们再也不用自己去操心dom了，只需要操心控制dom的数据就行了。那这样做的缺点是什么呢？那就是还有很多动态显示组件的功能做不到，不过NG一直在完善相关指令。NG4.0+给ngIf加入了else逻辑。

      可是接着又产生了更多的问题：为什么要使用这些东西？还要涉及到如此多的设计模式？

      ……

      好的，这个过程一直继续下去的话，我不仅可以接触大量TS(ES或者说JS)的知识，比如说类、继承、函数(包括箭头函数)、逻辑语句、运算符、优先级、数据类型、异步、模块化、DOM等知识，还可以了解很多设计模式、架构风格、性能优化、编码范式、注释风格等等非js层面的东西。

      js语言本身也在不断发展的。当自己以后碰见问题时能够快速想出解决方案的时候，用js还是ts甚至把甩给后端做都无所谓。

      所以说到底啊，不要只把眼光局限于js上，要放在整个前端开发上甚至编程思维上，为了提升js能力而提升js能力往往是达不成想要的效果的。

      古人有句话：求其上者得其中 求其中者得其下。

  • 虚拟DOM

  • 全面理解虚拟DOM，实现虚拟DOM

  • Vue.js 使用了基于 HTML 的模版语法，允许开发者声明式地将 DOM 绑定至底层 Vue 实例的数据。所有 Vue.js 的模板都是合法的 HTML ，所以能被遵循规范的浏览器和 HTML 解析器解析。

    在底层的实现上， Vue 将模板编译成虚拟 DOM 渲染函数。结合响应系统，在应用状态改变时， Vue 能够智能地计算出重新渲染组件的最小代价并应用到 DOM 操作上。

    如果你熟悉虚拟 DOM 并且偏爱 JavaScript 的原始力量，你也可以不用模板，直接写渲染（render）函数，使用可选的 JSX 语法。
